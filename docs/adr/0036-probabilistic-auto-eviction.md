# ADR-0034: Probabilistic Auto-Eviction for Storage Maintenance

## Status
Accepted

## Context
`beautyspot` は、キャッシュのTTL (Time-To-Live) をサポートしており、期限切れのデータは論理的に無効化され、次回のアクセス時に関数が再実行されます。
しかし、ユーザーが明示的に `MaintenanceService.clean_garbage()` を呼び出さない限り、有効期限切れのメタデータ（SQLite内のレコード）や、巨大な孤立したBlobファイル（S3やLocal Storage）は物理的に削除されず、長期間稼働するアプリケーションにおいてストレージが肥大化し続けるという問題がありました。

この問題を解決するため、ユーザーにインフラ管理を意識させない「黒子」としての哲学を維持しつつ、自動でガベージコレクション（エビクション）を行う仕組みが必要になりました。

## Options Considered

自動エビクションを実装するにあたり、以下の3つのアプローチを検討しました。

1.  **Capacity-based LRU (Least Recently Used) Policy:**
    * **概要:** ストレージ容量やDBの最大レコード数を設定し、上限に達した際に最も古いアクセス履歴を持つデータを削除する。
    * **評価:** 棄却。キャッシュヒットのたびに `last_accessed_at` などのタイムスタンプをDBに書き込む (UPDATE) 必要があり、メインスレッドのレイテンシ低下や SQLite の Write-Lock 競合を引き起こします。また、S3などのクラウドストレージにおける総容量の計算はAPIコールのコストが高すぎます。

2.  **Load-Adaptive Background Eviction:**
    * **概要:** バックグラウンドの保存キューの長さを監視し、システムが完全にアイドル状態のときにのみクリーンアップタスクを投入する。
    * **評価:** 棄却。実装が複雑になる（オーバーエンジニアリング）。`beautyspot` の主なユースケースにおいて、キューの厳密な状態管理を作り込むコストに見合うほどのメリットが得られないと判断しました。

3.  **Probabilistic Auto-Eviction (確率的自動実行):**
    * **概要:** キャッシュミスに伴う新しい結果の保存処理が発生した直後、設定された確率（例: 1%）に基づいて、バックグラウンドでクリーンアップタスクをエンキューする。
    * **評価:** **採用。**

## Decision

**Option 3 (Probabilistic Auto-Eviction)** を採用します。

* `Spot` の初期化パラメータに `eviction_rate: float = 0.0` を追加します。
* デフォルト値は `0.0`（無効）とし、意図しないデータの消失を防ぐオプトイン方式とします。
* エビクション処理（`clean_garbage`）はファイルI/Oを伴うため、メインスレッドや `asyncio` ループをブロックしないよう、別スレッドにオフロードして `_BackgroundLoop` に投入します。
* マルチスレッド環境での SQLite のロック競合や無駄な同時実行を防ぐため、`MaintenanceService.clean_garbage` の内部に非ブロッキングロック（Try-Lock）を導入し、既に他の掃除タスクが実行中の場合は安全にスキップ（無視）させます。

## Consequences

* **Positive:** ユーザーは `eviction_rate=0.01` のように設定するだけで、アプリケーションのメインロジックの応答速度（レイテンシ）を一切犠牲にすることなく、ストレージの肥大化を自動で防ぐことができます。
* **Positive:** 実装が極めてシンプルであり、バグの混入リスクや保守コストを低く抑えることができます。
* **Negative:** エビクションは確率に依存するため、「この関数を1回呼べば確実にストレージ容量が空く」といった決定論的な保証はありません。厳密な容量管理が必要なユーザーは、引き続き定期的なバッチスクリプト等で `clean_garbage()` を明示的に呼び出す必要があります。
