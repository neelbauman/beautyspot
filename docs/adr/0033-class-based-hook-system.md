# ADR 0033: ユーザー拡張のためのクラスベース・フックシステムと並列実行サポート

## ステータス

承認済 (Accepted)

## コンテキスト（背景と課題）

`beautyspot` のコアロジックを汚染することなく、ユーザーが「トークン計算」や「レイテンシ計測」などのカスタムメトリクスを収集できる仕組みが必要である。また、並列実行環境においても、競合状態を避けつつ安全に状態を共有・更新できることが求められる。

## 決定事項

### 1. クラスベースのフックインターフェース (`HookBase`)

状態（開始時間や累計カウンタ）を保持しやすいよう、クラスベースのインターフェースを採用する。

### 2. 並列実行のための自動ロック機構 (`ThreadSafeHookBase`)

`__init_subclass__` を利用し、サブクラスでオーバーライドされたメソッドを自動でロックラッパーで包む。これにより、ユーザーは明示的なロック制御なしでスレッドセーフな集計が可能になる。

### 3. 型安全なコンテキストオブジェクトの分離

単一の汎用コンテキストクラスではなく、実行フェーズごとに最適化された**3つの専用コンテキストクラス**を提供する。

* **`PreExecuteContext`**: 関数実行前の状態。戻り値（`result`）は存在せず、入力引数（`args`, `kwargs`）やキャッシュキーにアクセス可能。
* **`CacheHitContext`**: キャッシュ取得成功時の状態。ストレージから復元された `result` と `version` が含まれる。
* **`CacheMissContext`**: 関数実行完了時の状態。新たに生成された `result` と `version` が含まれる。

**設計の狙い:**

* **静的解析の最大化**: 存在しない属性（例：実行前に `result` を参照する）へのアクセスを IDE や MyPy が検知できるようにする。
* **関心の分離**: キャッシュヒット時のみ必要なメタデータなど、フェーズ固有の情報を明確に定義する。

### 4. ゼロオーバーヘッドの原則

フックが登録されていない場合は、コンテキストオブジェクトの生成を完全にスキップする。

## 結果（トレードオフ）

### ポジティブな影響

* **優れた DX (Developer Experience)**: 専用コンテキストにより、IDE でのプロパティ補完が正確に機能し、開発時の実行時エラーを未然に防げる。
* **並列処理の安全保障**: `ThreadSafeHookBase` により、複雑なマルチスレッドプログラミングの知識がなくても安全な拡張が可能になる。

### ネガティブな影響

* **実装の複雑性**: ライブラリ内部では複数のコンテキストクラスを管理する必要がある。
* **ボイラープレート**: ユーザーは `HookBase` 継承クラスを作成する手間があるが、型安全性と拡張性のメリットがこれを上回ると判断。
