# 0037: Bounded LRU Cache for Subclass Resolution in Serializer

## Context
`MsgpackSerializer` の `_default_packer` において、クラスの解決結果を保存する `_subclass_cache` 辞書が、動的に型が生成される環境（例: 実行時に生成される `namedtuple` や `pydantic` の動的モデルなど）で無制限に肥大化し、サイレントなメモリリークを引き起こす懸念が指摘された。
この辞書へのアクセスはシリアライズ処理のノードごとに高頻度で発生するため、標準の `threading.Lock` 等を用いて厳密なスレッドセーフティを担保しようとすると、深刻なパフォーマンスのボトルネック（ロック競合）を招いてしまう。

## Decision
外部依存を増やさず標準ライブラリの `collections.OrderedDict` を利用し、上限サイズ（デフォルト 1024）を持つ **LRU (Least Recently Used) キャッシュ** を導入する。
また、マルチスレッド環境下でのアクセスに対しては、意図的にロックフリーな（Optimisticな）アプローチを維持する。

## Consequences
* **Positive**: 動的な型生成が多用されるアプリケーションにおいても、キャッシュサイズが上限に保たれ、メモリ肥大化（メモリリーク）を完全に防止できる。
* **Positive**: シリアライズのホットパスにロックのオーバーヘッドが乗らないため、高いパフォーマンスを維持できる。
* **Negative**: スレッドセーフではないため、マルチスレッド環境下で同時に複数の新規クラスが解決された際、ごく稀にLRUの順序が乱れたり、重複した解決処理が走ったりする可能性がある。しかし、プロセスがクラッシュすることはなく、実用上のパフォーマンスインパクトも軽微であるため許容する。
