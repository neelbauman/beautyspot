# ADR 0035: GC時のデータロスト防止とバックグラウンドループのシャットダウン戦略

## 背景と課題 (Context)

`beautyspot` の `Spot` クラスは、非同期でのキャッシュ保存タスクを構造的に直列化するため、内部で専用の `asyncio` ループを実行するバックグラウンドスレッド (`_BackgroundLoop`) を保持している。
インスタンスのライフサイクル終了時、これらのバックグラウンドタスクを安全に終了させる必要があるが、シャットダウンのトリガー（明示的な呼び出し、アプリケーションの終了、ガベージコレクション）によって、メインスレッドへの影響とプロセスの振る舞いが大きく異なるという課題があった。

特に、ガベージコレクション (GC) はユーザーコードの予測不能なタイミングで発生する。ここでタスク完了を待機（スレッドを `join`）してしまうと、メインスレッドがフリーズしたりデッドロックを引き起こす危険性がある。一方で、待機せずにタスクを強制キャンセル（`wait=False`）すると、保存待ちだったキャッシュデータが失われてしまう。

## 決定事項 (Decision)

バックグラウンドループのシャットダウンにおいて、「メインスレッドのブロック有無」と「タスクの完了保証」のトレードオフを解決するため、以下の3つの明確なシャットダウン方式を実装し、状況に応じて厳密に使い分けることとする。

1. **`stop(wait=True)`** (完全同期シャットダウン)
* **挙動:** キューにある全タスクの完了を待機し、メインスレッドをブロック（`join`）する。
* **使い所:** ユーザーによる明示的な `shutdown()` 呼び出し、コンテキストマネージャの `__exit__`、およびプロセス終了時の `atexit` フック。
* **理由:** キャッシュの保存状態が「確定」したことをユーザーに保証するため。また、プロセス終了時 (`atexit`) には、メインスレッドをブロックしてプロセスを生き延びさせないと、OSによってスレッドごと強制終了されてしまうため。


2. **`stop_gracefully_no_wait()`** (非同期ドレイン / スレッド切り離し)
* **挙動:** メインスレッドは一切ブロックせず即座にリターンするが、バックグラウンドの `asyncio` ループには「現在積まれているタスクをすべて完了させてから自発的に終了せよ」という指示を出し、スレッドを切り離す（ゾンビ化させる）。
* **使い所:** `weakref.finalize` など、GC (ガベージコレクション) による破棄時のみ。
* **理由:** メインスレッドのフリーズ（デッドロック）を完全に回避しつつ、未保存のタスクをキャンセルせずに救済するため。


3. **`stop(wait=False)`** (強制停止)
* **挙動:** 実行中のタスクを強制キャンセルし、スレッドを即座に破棄する。
* **使い所:** エラー発生時の緊急停止や、テストコードでの強制クリーンアップ時。



## 結果 (Consequences)

* **堅牢性の向上:** ユーザーが `Spot` を一時的なスコープでインスタンス化し、GCの対象になった場合でも、キャッシュデータが失われる（データロスト）リスクがほぼ解消された。
* **安全性の担保:** GC発生時にメイン処理がブロックされることがなくなり、ユーザーアプリケーションのパフォーマンスや安定性に悪影響を与えない。
* **明確なメンタルモデル:** 開発者・メンテナーに対し、「いつ・どのシャットダウン方式を使うべきか」という明確な基準が提示された。
