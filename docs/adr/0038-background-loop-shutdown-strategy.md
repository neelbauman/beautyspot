# 0038: Background Loop Shutdown Strategy and Threading Model

## Context
バックグラウンドでの非同期IOタスク（キャッシュのディスク保存やDB書き込みなど）を処理するイベントループにおいて、単純に `daemon=True` スレッドを使用すると、メインスレッド終了時にプロセスが即座に強制終了（kill）され、書き込み中のデータ破損やキャッシュのロストが発生する危険性があった。
一方で、`daemon=False` に変更して `atexit` で待機を試みるアプローチでは、Pythonの内部仕様（`threading._shutdown()`）により、タスクがハングした場合にプロセス全体が永遠に終了しなくなるデッドロックの罠が存在した。また、非同期タスクのトラッキング（`asyncio.all_tasks`）に依存すると、スケジュールの隙間でタスクをロストする Check-then-Act 競合が生じることが判明した。

## Decision
真の耐障害性を持つ Graceful Shutdown を実現するため、以下のアーキテクチャを採用する：
1. **デーモンスレッドの維持**: スレッドは `daemon=True` で起動し、プロセス終了時の無限ハング（OSレベルのデッドロック）をOSの力で防ぐ。
2. **明示的なタスク追跡とロック**: `_lock` と `_active_tasks` カウンタを用いたステートマシンを導入し、`submit` 時と `atexit` 時の競合状態（Race Condition）を完全に排除する。
3. **Grace Period（猶予期間）の導入**: `atexit` フック内でメインスレッドをブロックし、タイムアウト付き（デフォルト5秒）で `_thread.join()` を実行することで、インフライトのIOタスクが完了するための猶予を与える。

## Consequences
* **Positive**: ユーザーが明示的な終了処理（`spot.close()` など）を意識することなく、プロセス終了時に安全にデータがディスクへ永続化される（優れたDeveloper Experienceの維持）。
* **Positive**: ネットワークエラー等でIOタスクが無限にスタックした場合でも、タイムアウト後は容赦なくプロセスが終了するため、アプリケーションがハングアップしない。
* **Negative**: プロセス終了時に、バックグラウンド処理が終わるまで最大 `drain_timeout` 秒（デフォルト5秒）のブロックが発生する。
