# 0039: Garbage Collection Strategy for Abandoned Temporary Files

## Context
Blobデータの保存時、アトミックな書き込みを実現するために `tempfile.mkstemp` を使用して一時ファイルを作成し、書き込み完了後に `os.replace` で正規のファイル名（`.bin`）にリネームする設計を採用している。
しかし、Windowsのアンチウイルスソフトやバックアッププロセスの介入により `os.replace` が失敗するエッジケースが存在する。この際、フォールバックとして `os.unlink` で一時ファイルの削除を試みるが、ファイルがロックされている場合はこれも失敗し、一時ファイルがストレージ上に永久に残留し続ける（サイレントなストレージリーク）問題が指摘された。

## Decision
アトミック書き込みのフェイルセーフとして、以下の機構を導入する：
1. **専用サフィックスの導入**: 一時ファイルの生成時に、標準のランダム名ではなく `.spot_tmp` という専用のサフィックスを付与し、追跡を容易にする。
2. **遅延ガベージコレクション (GC)**: `MaintenanceService.clean_garbage` のパイプラインに、一時ファイルの削除処理 (`LocalStorage.clean_temp_files`) を統合する。
3. **Grace Period (猶予期間) の設定**: 現在書き込み中の別プロセスのファイルを誤って削除しないよう、ファイルの最終更新時刻 (`mtime`) が指定時間（デフォルト24時間）を経過しているもののみを削除対象とする。

## Consequences
* **Positive**: アプリケーションのクラッシュや予期せぬファイルロックが発生した場合でも、後続の自動エビクション処理によってストレージが自動的に自己修復されるようになった。
* **Positive**: ユーザーはストレージリークを意識することなく、`beautyspot` を安全に運用できる。
* **Negative**: 失敗した一時ファイルは最大24時間ストレージの容量を占有し続けるが、実用上のパフォーマンスインパクトは皆無であると評価した。
