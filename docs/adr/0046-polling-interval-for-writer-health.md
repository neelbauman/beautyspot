# ADR 0046: SQLite ライタースレッドの死活監視におけるポーリング間隔の維持

## コンテキスト
`SQLiteTaskDB` は専用のライタースレッドを用いて書き込みを直列化しています。メインスレッドが書き込み完了を待機する際、または `flush` を実行する際、ライタースレッドが予期せず死亡（セグメンテーションフォールトや深刻なエラー）した場合にメインスレッドが永久にハングアップするのを防ぐ必要があります。

現在、`task.event.wait(timeout=0.5)` をループ内で使用し、定期的にスレッドの生存確認（`is_alive()`）を行っています。

## 意思決定
0.5秒というポーリング間隔を維持し、`threading.Condition` 等による純粋な通知ベースへの移行、あるいは極端な短縮を行わないことを決定しました。

## 根拠
1. **ハングアップの防止**: ライタースレッドが通知を送る前に死亡した場合でも、最大0.5秒以内に異常を検知してメインスレッドを解放できる「安全装置」として機能します。
2. **正常系パフォーマンス**: `threading.Event.wait` はイベントがセットされた瞬間に即座に復帰するため、書き込みが成功している限り、この 0.5 秒という数値はレイテンシに一切影響しません。
3. **リソース効率**: ポーリング間隔を極端に短く（例: 0.01秒）すると、重い I/O を待機している間も頻繁にコンテキストスイッチが発生し、電力効率や CPU リソースを浪費します。
4. **シンプルさ**: 監視専用のスレッドを別途導入するよりも実装が簡潔であり、デッドロックのリスクが低い設計です。

## トレードオフ
- **異常検知の遅延**: スレッドが死亡した際、メインスレッドがエラーを認識するまでに最大 0.5 秒のタイムラグが発生します。しかし、通常のアプリケーションにおいて、この稀な異常事態における 0.5 秒の遅延は許容範囲内であると判断しました。
