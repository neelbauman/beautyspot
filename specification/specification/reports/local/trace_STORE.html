ARCH004 ARCH STORE ✓ レビュー済

REQ004「大きなキャッシュデータを外部ストレージに保存できること、DB直接保存とBlob保存を自動判定できること」を実現するためのアーキテクチャを定義する。

二層保存モデル

キャッシュデータの保存先を「DB直接（インライン）」と「外部Blobストレージ」の2層に分離する。小さなデータはDBに直接保存してI/Oオーバーヘッドを回避し、大きなデータは外部ストレージに保存してDBの肥大化を防止する。

  • DIRECT_BLOB: シリアライズ済みデータをDBの result_data カラムにインライン保存する。
  • FILE: シリアライズ済みデータを外部Blobストレージに保存し、DBにはロケーション（ファイルパスやS3 URI）を記録する。

ストレージバックエンドの抽象化

外部ストレージのバックエンドを抽象インターフェース (BlobStorageBase) で定義し、ローカルファイルシステムやクラウドオブジェクトストレージ（S3等）を差し替え可能にする。ファクトリ関数がURIスキームに基づいて適切な実装を自動選択する。

ポリシーベースの保存先判定

DB直接保存とBlob保存の切り替えを、差し替え可能なポリシーオブジェクト (StoragePolicyProtocol) に委譲する。ユーザーは関数単位で明示的に保存先を指定することもでき、以下の優先順位で判定する:

  1. @mark(save_blob=True/False) による明示指定（最優先）
  2. StoragePolicyProtocol.should_save_as_blob(data) によるポリシー判定
  3. デフォルトポリシー (WarningOnlyPolicy: 常にDB直接保存、閾値超過時に警告のみ)

SPEC009 SPEC STORE ✓ レビュー済

LocalStorage はローカルファイルシステムにBlobを保存する BlobStorageMaintenable 実装である。

  • save(key, data): tempfile.mkstemp で一時ファイルを作成し、flush + fsync の後 os.replace でアトミックにリネームする。並行書き込みやクラッシュ時のファイル破損を防止する。キーにパス区切り文字 (.., /, \\) を含む場合は ValidationError を送出する。
  • load(location): 解決パスが base_dir 配下であることを is_relative_to で検証し、パストラバーサルを防止する。ファイル不在時は CacheCorruptedError を送出する。
  • delete(location): 冪等に動作する。空ディレクトリの削除はメンテナンスタスクに委譲する。
  • list_keys(): .bin ファイルを rglob で再帰的に列挙し、base_dir からの相対パスを返す。
  • get_mtime(location): パストラバーサル検証後にファイルの最終更新時刻を返す。
  • clean_temp_files(max_age_seconds): 猶予期間（デフォルト24時間）を超えた .spot_tmp ファイルを清掃する。
  • prune_empty_dirs(): 空ディレクトリ（.DS_Store 等のみ含むものを含む）を再帰的に削除する。base_dir 自体は保持する。
  • 初期化時に base_dir を絶対パスに正規化し、ディレクトリと .gitignore を自動作成する。

SPEC010 SPEC STORE ✓ レビュー済

S3Storage はAWS S3互換オブジェクトストレージにBlobを保存する BlobStorageMaintenable 実装である。

  • save(key, data): boto3.upload_fileobj によるストリームアップロードを行い、5GB超のマルチパートアップロードに自動対応する。保存先を s3://bucket/prefix/key.bin 形式のURIとして返す。
  • load(location): s3:// URIをパースし get_object でデータを取得する。オブジェクト不在時は CacheCorruptedError を送出する。
  • delete(location): delete_object で削除する。権限エラー等の深刻な障害はログに記録し、後続のGCに委譲する。
  • list_keys(): list_objects_v2 をページネーション付きで列挙し、s3:// URIを返す。
  • get_mtime(location): head_object の LastModified をPOSIXタイムスタンプとして返す。
  • コンストラクタは s3://bucket/prefix 形式のURIからバケット名・プレフィックスを分離する。プレフィックス省略時は blobs をデフォルトとする。
  • boto3 がインストールされていない場合はインスタンス生成時に ImportError を送出する。

SPEC011 SPEC STORE ○ 未レビュー

StoragePolicyProtocol は should_save_as_blob(data: bytes) -> bool を定義し、シリアライズ済みデータのサイズに基づいてBlob保存の要否を判定する。

実装3種:

- ThresholdStoragePolicy(threshold): データサイズが閾値を超過した場合に True を返す。自動最適化の推奨ポリシー

- WarningOnlyPolicy(warning_threshold): 常に False を返す（DB直接保存）が、閾値超過時に警告をログ出力する。v2.0互換のデフォルトポリシー

- AlwaysBlobPolicy(): 常に True を返す。全データをBlobストレージに保存する。

SPEC024 SPEC STORE ✓ レビュー済

ARCH004で定義された「ストレージバックエンドの抽象化」を実現する具体的なインターフェース仕様。

  • BlobStorageBase (ABC): save, load, delete, list_keys, get_mtime の5メソッドを定義する。新しいストレージバックエンドはこのクラスを継承して実装する。
  • BlobStorageCore (Protocol): 実行時に必須の3メソッド (save, load, delete) を定義する最小インターフェース。
  • Maintenable (Protocol): メンテナンス操作に必要な2メソッド (list_keys, get_mtime) を定義する拡張インターフェース。
  • BlobStorageMaintenable: BlobStorageCore と Maintenable の合成プロトコル。
  • create_storage(path, options): URIスキーム (s3://) に基づいて適切な実装を自動選択するファクトリ関数。s3:// で始まる場合は S3Storage、それ以外は LocalStorage を返す。
